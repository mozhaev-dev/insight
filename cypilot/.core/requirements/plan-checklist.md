## Overview

This checklist validates execution plans generated by the `plan` workflow. It ensures plans are complete, self-contained, and will execute correctly across multiple agent sessions.

**Critical focus areas**:
- All interactive questions from source rules are preserved in phase files
- All kit rules are covered without omission
- Post-completion handoff is properly configured
- Validation step is included as mandatory next action

---

## When to Use

- **After plan generation** — MANDATORY validation before execution
- **During plan analysis** — when user invokes `/cypilot-analyze` on a plan
- **Plan debugging** — when phases fail or produce unexpected results

---

## Checklist Categories

### 1. Structural Validation

**Plan manifest (`plan.toml`)**:

- [ ] `plan.toml` exists in `.plans/{task-slug}/`
- [ ] `[plan]` section contains: `task`, `type`, `target`, `created`, `total_phases`, `lifecycle`
- [ ] `[[phases]]` blocks match actual phase files
- [ ] Phase numbers are sequential (1, 2, 3, ...)
- [ ] `depends_on` arrays form valid DAG (no cycles)
- [ ] `outputs` and `inputs` arrays are consistent across phases (output of phase N appears in inputs of phase M where M depends on N)

**Brief files**:

- [ ] Each `[[phases]]` entry has `brief_file` field
- [ ] All brief files exist: `brief-{NN}-{slug}.md` for each `[[phases]]` entry
- [ ] Each phase file has a corresponding brief file (1:1 mapping)

**Phase files**:

- [ ] All phase files exist: `phase-{NN}-{slug}.md` for each `[[phases]]` entry
- [ ] Each phase file has valid TOML frontmatter
- [ ] Frontmatter `number` matches filename
- [ ] Frontmatter `total` matches `total_phases` in plan.toml
- [ ] All 9 sections present in correct order: Preamble, What, Prior Context, [User Decisions], Rules, Input, Task, Acceptance Criteria, Output Format

---

### 1b. Brief-Phase Integrity

> **⛔ CRITICAL**: Skipping briefs is the #1 process violation in plan generation. Briefs enforce the contract between decomposition and compilation — without them, phase files are compiled from accumulated context and silently drop kit content.

- [ ] Every brief file written to disk before phase compilation started
- [ ] Each brief's Load Instructions reference specific kit file line ranges
- [ ] Each phase file's Rules section covers the line ranges from its brief
- [ ] No phase file contains kit content not referenced in its brief (no "extra" inlined content)

---

### 2. Interactive Questions Completeness

> **⛔ CRITICAL**: This is the #1 source of plan quality failures. Every interactive question from source rules MUST appear in the compiled plan.

**Source scanning** (performed during plan generation):

- [ ] Target workflow file scanned for interaction patterns
- [ ] `rules.md` for target artifact kind scanned for interaction patterns
- [ ] `checklist.md` for target artifact kind scanned for interaction patterns
- [ ] `template.md` for target artifact kind scanned for interaction patterns
- [ ] ALL files referenced by `ALWAYS open`, `OPEN and follow` directives scanned

**Interaction pattern detection**:

| Pattern Type | Regex/Keywords | Example |
|-------------|----------------|---------|
| Questions | `ask the user`, `ask user`, `what is`, `which`, `?` at end of instruction | "Ask the user which modules to include" |
| Input requests | `user provides`, `user specifies`, `user enters`, `input from user` | "User provides project name" |
| Confirmation gates | `wait for`, `confirm`, `approval`, `before proceeding` | "Wait for user confirmation" |
| Review requests | `review`, `present for`, `show to user`, `user inspects` | "Present output for user review" |
| Decision points | `choose`, `select`, `option A or B`, `decide` | "User selects architecture style" |

- [ ] All detected interaction points listed in plan generation output
- [ ] Each interaction point classified: `pre-resolvable`, `phase-bound`, or `cross-phase`
- [ ] Pre-resolvable questions asked BEFORE plan generation
- [ ] Cross-phase questions asked BEFORE plan generation
- [ ] Phase-bound questions embedded in appropriate phase files

**Phase file verification**:

- [ ] Each phase with interaction points has `## User Decisions` section
- [ ] Pre-resolved decisions listed under `### Already Decided`
- [ ] Phase-bound questions listed under `### Decisions Needed During This Phase`
- [ ] Review gates appear as explicit Task steps: "Present output to user for review"
- [ ] No interaction point from source rules is missing from compiled phases

**Completeness check formula**:

```
interaction_points_in_plan = sum(questions in all phase files)
interaction_points_in_sources = scan(workflow + rules + checklist + template + all OPEN/ALWAYS references)

PASS if: interaction_points_in_plan >= interaction_points_in_sources
FAIL if: any source interaction point is not represented in plan
```

**Content placement check formula**:

```
for each phase:
  # Kit rules MUST be inlined (filtered, not raw-copied)
  FAIL if: Rules section references rules.md by path instead of inlining
  FAIL if: Rules section contains Prerequisites / Load Dependencies (these are load directives)
  FAIL if: Rules section contains Tasks from rules.md (phase has its own Task section)
  FAIL if: Rules section is missing MUST/MUST NOT requirements from rules.md
  
  # Kit template/checklist content MUST be inlined for assigned sections
  FAIL if: Input section references checklist.md by path instead of inlining
  FAIL if: Input section references template.md by path instead of inlining
  
  # Project content MUST NOT be inlined
  FAIL if: Input section contains full content of project artifacts
  FAIL if: Input section contains source code (except small snippets as examples)
  
  # Runtime reads MUST have Task instructions
  for each file in input_files:
    FAIL if: no corresponding "Read {file}" step in Task section
  for each file in inputs:
    FAIL if: no corresponding "Read {file}" step in Task section
```

**Context budget check**:

```
for each phase:
  estimate = phase_file_lines + sum(input_files lines) + sum(inputs lines) + output_lines
  FAIL if: estimate > 2000 lines (OVERFLOW)
  WARN if: estimate > 1500 lines
```

### 3. Rules Coverage

> **⛔ CRITICAL**: Kit rules are LAW. Every MUST/MUST NOT rule must appear verbatim in at least one phase.

**Coverage verification**:

- [ ] `rules.md` for target artifact kind read and filtered
- [ ] Every MUST rule from `rules.md` appears in at least one phase's Rules section
- [ ] Every MUST NOT rule from `rules.md` appears in at least one phase's Rules section
- [ ] Rules inlined VERBATIM — not summarized, trimmed, or paraphrased
- [ ] Stripped from Rules section: Prerequisites, Load Dependencies, Tasks, Next Steps, TOC (these are workflow directives, not rules)
- [ ] Union of all phases' Rules sections covers 100% of applicable MUST/MUST NOT rules

**For analyze plans**:

- [ ] `checklist.md` for target artifact kind fully loaded
- [ ] Every checklist item appears in at least one phase
- [ ] Checklist items are inlined VERBATIM

**Verification procedure**:

1. Extract all MUST/MUST NOT statements from source `rules.md`
2. For each statement, search all phase files' Rules sections
3. Mark as COVERED if found verbatim, MISSING if not found
4. Report coverage percentage and list missing rules

---

### 4. Context Completeness

**Navigation rules processed**:

- [ ] Target workflow file opened and scanned for navigation directives
- [ ] All `ALWAYS open` directives processed
- [ ] All `OPEN and follow` directives processed
- [ ] All `ALWAYS open and follow` directives processed
- [ ] Manifest of loaded files recorded and reported to user

**Variable resolution**:

- [ ] `cpt info` or `cpt resolve-vars` executed
- [ ] All `{variable}` references resolved to absolute paths
- [ ] No unresolved `{...}` patterns in phase files (outside code fences)

**Kit content inlined** (stable, compile-time — filtered per compilation brief):

- [ ] Kit rules (`rules.md`) MUST/MUST NOT requirements inlined in Rules section
- [ ] Kit rules Prerequisites/Tasks/Next Steps STRIPPED (not rules, just workflow directives)
- [ ] Kit checklist (`checklist.md`) criteria inlined in Input section (assigned sections only)
- [ ] Kit template (`template.md`) sections inlined in Input section (assigned sections only)
- [ ] Kit examples excerpted where helpful
- [ ] No references to kit files by path ("see rules.md", "load checklist")

**Project content read at runtime** (dynamic):

- [ ] Project artifacts (PRD, DESIGN, FEATURE) listed in `input_files`, NOT inlined
- [ ] Source code files listed in `input_files`, NOT inlined
- [ ] Intermediate results (`out/*.md`) listed in `inputs`, NOT inlined
- [ ] Task section contains explicit read instructions for each `input_files` entry
- [ ] Task section contains explicit read instructions for each `inputs` entry

---

### 5. Phase Independence

**Self-containment**:

- [ ] Each phase file executable without Cypilot knowledge
- [ ] Each phase file executable without reading other phase files
- [ ] Prior Context section summarizes (not references) earlier outputs
- [ ] All tool commands include full arguments (no "use same args as before")

**Intermediate results**:

- [ ] Phases that produce data for later phases declare `outputs` in frontmatter
- [ ] Phases that consume prior data declare `inputs` in frontmatter
- [ ] `outputs` paths use `out/` directory for intermediate files
- [ ] Final phase's inputs list includes all necessary prior outputs

---

### 6. Budget Compliance

**Phase file size**:

- [ ] Each phase file ≤ 1000 lines (hard limit)
- [ ] Each phase file ≤ 500 lines (target)
- [ ] Rules section ≤ 300 lines per phase
- [ ] Input section ≤ 500 lines per phase
- [ ] Task section has 3-10 steps

**Execution context prediction**:

- [ ] Each phase's execution context estimated
- [ ] No phase exceeds 5000 lines predicted execution context (OVERFLOW)
- [ ] Phases in WARNING zone (3001-5000 lines) noted
- [ ] OVERFLOW phases (>5000) auto-split before compilation

---

### 7. Lifecycle & Handoff

> **⛔ CRITICAL**: LLMs forget to ask about plan lifecycle. This MUST be enforced.

**Plan lifecycle**:

- [ ] User asked about lifecycle strategy after plan generation
- [ ] `lifecycle` field set in `plan.toml`: `gitignore`, `cleanup`, `archive`, or `manual`
- [ ] Lifecycle action implemented (gitignore added, cleanup phase added, etc.)

**Post-completion handoff**:

- [ ] Last phase's Output Format includes `ALL PHASES COMPLETE` block
- [ ] Last phase's Output Format includes lifecycle strategy reference
- [ ] Last phase's Output Format asks `Continue in this chat? [y/n]`

**Next-phase prompts**:

- [ ] Each non-final phase's Output Format includes next-phase prompt
- [ ] Next-phase prompt is inside a single code fence (copy-pasteable)
- [ ] Next-phase prompt includes plan.toml path
- [ ] Next-phase prompt includes next phase file path

**Validation next-step** (after plan generation):

- [ ] User informed that plan should be validated before execution
- [ ] Validation command provided: `/cypilot-analyze` on plan directory
- [ ] Validation offered as explicit next step option

---

## Validation Procedure

**For plan self-validation (after generation)**:

1. Run through all checklist categories above
2. For each FAIL, report the specific issue and location
3. Calculate pass rate: `passed_items / total_items`
4. If pass rate < 100%, plan MUST be corrected before execution

**For plan analysis (`/cypilot-analyze` on plan)**:

1. Load plan.toml and all phase files
2. Load source rules, checklist, template for target artifact kind
3. Run through all checklist categories
4. Output analysis report with per-category status

**Automated checks** (can be done by `cpt` command):

- Structural validation (files exist, TOML valid, sections present)
- Variable resolution (no unresolved `{...}`)
- Budget compliance (line counts)

**Manual checks** (require LLM reasoning):

- Interactive questions completeness (semantic matching)
- Rules coverage (verbatim matching)
- Context completeness (navigation rule processing)
